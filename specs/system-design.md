# System Design: Provenance Guard

**AI201 Project 4 — Attribution, Confidence & Production Safety**

---

## What Provenance Guard Does

Provenance Guard is a backend a creative-sharing platform can plug in to answer one question
about a piece of submitted text: *was this written by a person, or generated by an AI tool?* It
does not give a yes/no verdict. It runs the text through two independent detection signals,
combines them into a single calibrated confidence score, turns that score into a plain-language
transparency label for the reader, logs the decision, and gives the creator a way to appeal.

This matters because perfect AI detection is an unsolved problem, and on a writing platform the
costs are asymmetric: falsely accusing a human of passing off AI work is far worse than missing a
piece of AI text. The system is therefore built to *communicate uncertainty honestly* rather than
force a binary call, and to give creators recourse when it gets things wrong.

## Pipeline

### Submission

```
Submitted text  +  creator_id
      │
      ▼
POST /submit  (rate limited: 10/min, 100/day)
      │
      ▼  content_id assigned (uuid)
      │
      ├──▶ llm_signal()          ──▶ {"p_ai": float, "rationale": str}   (semantic, Groq)
      │
      ├──▶ stylometric_signal()  ──▶ {"style_score": float, "reliable": bool, "metrics": {…}}  (structural)
      │
      ▼
combine_signals()  ──▶  {"p_ai": float, "attribution": "likely_ai" | "uncertain" | "likely_human"}
      │
      ▼
make_label()       ──▶  transparency label text (varies by band)
      │
      ├──▶ log_submission()  ←── side effect: appends to logs/audit.jsonl
      │
      ▼
Response: {content_id, attribution, confidence, label}
```

### Appeal

```
content_id  +  creator_reasoning
      │
      ▼
POST /appeal
      │
      ▼  find_submission(content_id)  ──▶ 404 if unknown
      │
      ▼
log_appeal()  ──▶ appends an "appeal" record with status "under_review" beside a snapshot of
      │           the original decision (side effect: logs/audit.jsonl)
      ▼
Response: {content_id, status: "under_review", message}
```

All stages are required for the system to behave correctly. A classifier without calibrated
scoring produces overconfident labels. A label without an appeal path leaves a misclassified
creator with no recourse. A pipeline without a log is unaccountable.

## The Confidence Model

The system's core output is `p_ai` — its estimated probability, in `[0, 1]`, that the text is
AI-generated. The score is mapped to three attribution bands:

| Band | Range (`p_ai`) | Meaning | Label shown |
|------|----------------|---------|-------------|
| `likely_ai` | `≥ 0.80` | Strong evidence of AI generation | High-confidence AI |
| `uncertain` | `0.30 – 0.80` | Signals weak or in conflict | Uncertain |
| `likely_human` | `≤ 0.30` | No strong signs of AI generation | High-confidence human |

### Why a score, not a binary?

A binary AI/human flag throws away the most important information: *how sure the system is*. A
0.51 and a 0.95 are completely different situations — one is a coin flip, the other is strong
evidence — but a binary flag collapses them into the same "AI" stamp. A continuous score lets the
label honestly distinguish "we're confident" from "we're guessing," which is the whole point of a
transparency feature.

### Why the bands are asymmetric

The `likely_ai` band starts at `0.80`, not `0.50`. This is deliberate. A **false positive** — a
human's work labeled AI — is the worst outcome on a writing platform, so the system requires
strong evidence before it will assert "AI." Everything borderline degrades to `uncertain` rather
than to an accusation. The `uncertain` band is wide on purpose: when the system isn't sure, it
says so.

### The caution of the middle band

`uncertain` is the most consequential output, and the one the system produces most often for
real-world prose, because genuine human and AI writing overlap heavily. The design treats this as
a feature, not a failure: an honest "we can't tell" plus an appeal path is more trustworthy than a
confident wrong answer.

## The Two Signals

Detection uses **two genuinely independent signals** so they fail in different situations.
Agreement between them is real evidence; disagreement is itself a useful uncertainty signal.

| Signal | File / function | What it captures | Type |
|--------|-----------------|------------------|------|
| 1 | `detector.llm_signal()` | Holistic "does this read as AI" — tone, voice, fluency, over-uniform structure | Semantic (Groq LLM) |
| 2 | `detector.stylometric_signal()` | Structural uniformity — sentence-length burstiness, vocabulary diversity, punctuation rate | Statistical (pure Python) |

Signal 1 is an *LLM-as-judge* call: a language model evaluating input rather than generating
end-user output. Signal 2 is hand-computed statistics with no API call. Because one is semantic
and one is structural, the stylometric blind spot (formal, uniform human writing) is exactly where
the LLM's holistic read can rescue the verdict, and vice-versa. Single-signal detection would have
no such cross-check.

## Components

**`app.py`** — Flask orchestration and the API surface: `POST /submit` (rate limited), `POST
/appeal`, `GET /log`. Calls the detector, the labeler, and the auditor in order and shapes the JSON
response.

**`detector.py`** — Both detection signals and the confidence scoring. `llm_signal()`,
`stylometric_signal()`, `combine_signals()`, and `score_to_attribution()`. *(Milestones 3–4.)*

**`labeler.py`** — `make_label()` maps an attribution to one of three reader-facing label strings.
*(Milestone 5.)*

**`auditor.py`** — Structured audit logging and content-status tracking. `log_submission()`,
`log_appeal()`, `find_submission()`, `get_log()`. *(Milestones 3 & 5.)*

**`config.py`** — Constants: `GROQ_API_KEY`, `GROQ_MODEL`, `LOG_PATH`, the score thresholds
(`AI_THRESHOLD`, `HUMAN_THRESHOLD`), the signal weights (`LLM_WEIGHT`, `STYLE_WEIGHT`),
`VALID_ATTRIBUTIONS`, and `SUBMIT_RATE_LIMIT`.

## Comparison to Production Detection Systems

The pattern here — multi-signal detection → calibrated score → user-facing disclosure → appeal — is
how real AI-content detection is approached, and where it gets hard.

- **Commercial detectors** (GPTZero, Turnitin's AI writing indicator, Originality.ai) all combine
  multiple signals and report a *probability or percentage*, not a binary, precisely because binary
  detection produces unacceptable false-positive rates on human writing.
- **Watermarking** (e.g. provider-side statistical watermarks) is a fundamentally different, more
  reliable signal — but it only works for text from cooperating providers, so detectors still need
  signal-based estimation for everything else.
- **The false-positive problem is real and documented:** detectors have flagged non-native English
  writing and formal human prose as AI, with serious consequences for the writers. That failure
  mode is exactly the one this system's asymmetric thresholds and appeal path are designed around.

The honest framing — which this system adopts — is that detection is *probabilistic and
contestable*, not authoritative.

## How This Builds on Lab 4 (RepairSafe)

- **LLM-as-judge classifier → Detection signal 1.** RepairSafe's `classify_safety_tier()` and this
  project's `llm_signal()` are the same architecture (Groq call, structured output, parse, validate),
  applied to a different judgement.
- **`.jsonl` audit log → Audit log.** Same append-only, one-JSON-object-per-line format and the same
  `[LOGGED]` console convention.
- **Spec-before-code.** RepairSafe shipped pre-written specs with blank fields; here the specs and
  `planning.md` were written from scratch before implementation.

What is new in this project: a second, non-LLM signal; continuous confidence scoring instead of a
discrete tier; and the production layer — transparency labels, an appeals workflow, and rate
limiting on a Flask API.
